High Availability Patterns

Multi-AZ architectures, load balancer selection, Auto Scaling policies, stateless design

Core Principles

  • Eliminate single points of failure — redundancy at every tier
  • Design for failure — assume any component can fail at any time without warning
  • Decouple layers — failure in one tier must not cascade to others
  • Automate recovery — health checks, Auto Scaling, self-healing managed services

Multi-AZ vs Multi-Region

Multi-AZMulti-Region
Protects againstAZ outage (data center failure)Regional outage
LatencyLow — same metro areaHigher — inter-region
ComplexityLowHigh (data consistency, routing)
CostLow–moderateHigh
Use forStandard HA (99.9%+)DR, global applications, data sovereignty

Auto Scaling Policies

PolicyBest For
Target TrackingSteady-state workloads; simplest to configure; e.g., keep CPU at 60%
Step ScalingRespond with different magnitudes to different alarm thresholds
Scheduled ScalingPredictable peaks — pre-scale the group before demand hits (eliminates warmup lag)
Predictive ScalingML-based forecast; pre-scales based on historical patterns; best combined with Target Tracking

Stateless vs Stateful Design

  • Make web and app tiers stateless: store session state externally in ElastiCache (Redis) or DynamoDB
  • Stateless instances can be terminated and replaced without data loss — enables clean Auto Scaling
  • ALB sticky sessions (session affinity) are an HA anti-pattern — avoid unless the application absolutely requires it

Health Checks

  • ALB health checks: HTTP/HTTPS to target; unhealthy targets removed from rotation immediately
  • Auto Scaling: default EC2 status checks; upgrade to ELB health checks for more thorough application-level checking
  • Route 53 health checks: endpoint polling; used with failover routing to detect regional failures

Reference Architectures

Standard 3-Tier Web App (Multi-AZ)

Route 53 (failover / latency routing)
  → CloudFront (optional edge caching)
    → ALB (multi-AZ, managed HA)
      → EC2 Auto Scaling Group (min 2 AZs)
        → RDS Aurora Multi-AZ  (or Aurora Global for multi-region)
        → ElastiCache Redis (Multi-AZ, automatic failover)

Serverless HA

Route 53 / CloudFront
  → API Gateway (regional, inherently multi-AZ)
    → Lambda (AWS-managed multi-AZ, 1,000 concurrent default)
      → DynamoDB (multi-AZ by default; Global Tables for multi-region)

Decoupling for Resilience

  • SQS: buffer between producer and consumer; consumer failure doesn't affect producer; DLQ captures unprocessable messages
  • SNS fan-out: one event triggers multiple consumers in parallel; each consumer fails independently
  • EventBridge: loose coupling via events; producers and consumers have no direct dependency
  • "Highly available" on the exam = survive an AZ failure at minimum
  • ALB + Auto Scaling Group across 2+ AZs is the canonical HA compute pattern
  • Lambda is inherently multi-AZ — no HA configuration needed
  • S3 provides 11-nines durability across 3 AZs by default — no HA configuration needed
  • ElastiCache Redis Multi-AZ: primary + replica in separate AZs; automatic failover in ~30–60s
