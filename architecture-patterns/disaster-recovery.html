Disaster Recovery Patterns

RPO/RTO tradeoffs, four DR strategies, Route 53 routing policies, AWS services by DR role

Key Metrics

  • RPO (Recovery Point Objective) — maximum acceptable data loss measured in time. "How old can the restored data be?"
  • RTO (Recovery Time Objective) — maximum acceptable downtime. "How long can we be unavailable?"

Lower RPO/RTO = more complex architecture = higher cost. The exam tests your ability to match a strategy to a given RPO/RTO + budget constraint.

DR Strategies

1. Backup & Restore RPO: hoursRTO: hoursCheapest

  • Periodic backups to S3/Glacier; infrastructure rebuilt from scratch on failure
  • Use CloudFormation for infra provisioning, AWS Backup for data
  • No standby infrastructure costs; longest recovery time

2. Pilot Light RPO: minutesRTO: 30–60 min

  • Core data services always running in DR region (e.g., database replication); compute is off
  • On DR event: provision compute (EC2/ASG), update DNS to DR region
  • Aurora Global Database or RDS cross-region replica keeps data warm

3. Warm Standby RPO: seconds–minutesRTO: minutes

  • Full but scaled-down functional environment running continuously in DR region
  • On DR event: scale up ASG/database to production capacity, shift traffic via Route 53
  • Costs more than pilot light but RTO is much lower — no infrastructure provisioning needed

4. Active-Active RPO: near zeroRTO: near zeroMost Expensive

  • Full production capacity running simultaneously in two+ regions
  • Route 53 latency routing or Global Accelerator distributes traffic
  • Requires application to handle concurrent multi-region writes (DynamoDB Global Tables, Aurora Global DB)

Decision Tree

What are your RPO and RTO requirements?

  RPO < 1 min  AND  RTO < 1 min?
    → Active-Active (multi-site)

  RPO < 1 min,  RTO < 10 min?
    → Warm Standby

  RPO < 15 min, RTO < 1 hr?
    → Pilot Light

  Cost is primary concern, hours of downtime acceptable?
    → Backup & Restore

Route 53 Routing Policies for DR

  • Failover: active-passive; primary serves traffic; health check triggers cutover to secondary
  • Latency: routes each user to the lowest-latency region (active-active)
  • Weighted: split traffic by percentage — use for canary deploys or gradual failover
  • Geolocation: route by country/continent — use for data sovereignty
  • Geoproximity: route by geographic distance with Traffic Flow bias
  • Calculated health checks: aggregate multiple child health checks into one parent
  • Route 53 health check propagation: ~30–60 seconds — factor this into your RTO calculations
  • Aurora Global Database managed failover is automatic (< 1 min); manual failover requires explicit API/console action
  • S3 CRR replicates new objects only — use S3 Batch Replication for existing objects
  • DynamoDB Global Tables: conflicts resolved by "last writer wins" — design for idempotency
  • RDS cross-region read replicas are asynchronous — non-zero RPO; suitable for pilot light/warm standby only
