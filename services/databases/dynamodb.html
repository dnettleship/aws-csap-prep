Amazon DynamoDB

Key-value & document store, indexes, DAX, Global Tables, Streams, capacity planning

Data Model

  • Partition key (PK) — required; determines physical partition; must be unique per item (when no sort key)
  • Sort key (SK) — optional; combined with PK forms a composite primary key; enables range queries within a partition
  • Item — a row; up to 400 KB; attributes are schema-free except for primary key
  • Hot partition — if all reads/writes hit one PK value, throughput is limited to that partition's capacity; design for uniform distribution

Read Consistency

  • Eventually consistent reads (default) — reads may return stale data for up to ~1 second; consumes 0.5 RCU per 4 KB
  • Strongly consistent reads — always returns the latest data; consumes 1 RCU per 4 KB; higher latency; not available on GSIs
  • Transactional reads (TransactGetItems) — consumes 2 RCUs per 4 KB; all-or-nothing across up to 25 items

DynamoDB Accelerator (DAX)

  • In-memory cache purpose-built for DynamoDB; fully API-compatible — change endpoint, not code
  • Reduces read latency from milliseconds to microseconds
  • Deployed in a VPC; Multi-AZ cluster with primary + replica nodes
  • Item cache (individual item reads) + query cache (Query/Scan results)
  • Not suitable for strongly consistent reads (DAX caches eventually-consistent data) or write-heavy workloads

DynamoDB Streams

  • Ordered log of item-level changes (create, update, delete) in a table; retained 24 hours
  • Stream record can contain: keys only, new image, old image, or both images
  • Common pattern: Streams → Lambda for event-driven processing, change data capture, cross-region replication (before Global Tables)
  • Kinesis Data Streams for DynamoDB: alternative to native Streams; retains data up to 1 year

Global Tables

  • Multi-region, multi-active replication — reads and writes in each region
  • Conflict resolution: last-writer-wins based on timestamp — design for idempotency
  • Requires on-demand capacity or provisioned with Auto Scaling enabled
  • Replication latency typically sub-second
  • Use for: global applications requiring local latency + regional failover with near-zero RPO

Other Key Features

  • TTL (Time to Live) — set an expiry epoch timestamp attribute; DynamoDB deletes expired items automatically within 48 hours; no capacity consumed
  • Point-in-Time Recovery (PITR) — continuous backups for 35 days; restore to any second in the window
  • Transactions — TransactWriteItems / TransactGetItems for ACID operations across up to 25 items in one or more tables
  • Conditional writes — optimistic locking; only write if condition is met (e.g. attribute doesn't exist)
  • Encryption at rest — enabled by default with AWS-owned keys; can use KMS CMK

  • LSIs must be created at table creation — cannot add them later
  • DynamoDB Global Tables conflict resolution is last-writer-wins — not suitable for debit/credit operations without application-level conflict handling
