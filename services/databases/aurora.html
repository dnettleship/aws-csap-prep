Databases

Amazon Aurora

MySQL & PostgreSQL-compatible — 5× MySQL throughput, 3× PostgreSQL, AWS-engineered storage

Architecture

  • Cluster: 1 primary (read/write) + up to 15 read replicas
  • Storage: 6-way replication across 3 AZs; auto-grows to 128 TiB; no manual provisioning
  • Endpoints:
    • Cluster endpoint → always points to primary (write)
    • Reader endpoint → load-balanced read traffic across replicas
    • Custom endpoints → defined subset of instances (e.g., analytics replicas)
    • Instance endpoints → connects to one specific instance

Key Features

Aurora Serverless v2

  • Scales in fine-grained 0.5 ACU increments within seconds
  • Minimum 0.5 ACU — does not scale to zero (use v1 for that)
  • Use for: variable workloads, dev/test, multi-tenant SaaS

Aurora Global Database

  • 1 primary region + up to 5 secondary read-only regions
  • Replication lag typically < 1 second
  • Managed failover: promote secondary in < 1 minute
  • Use for: global low-latency reads + cross-region DR

Aurora Multi-Master (MySQL only)

  • Multiple read/write nodes; conflict resolution happens at storage layer
  • Single-region only; use for continuous write availability, not write scaling

Aurora Parallel Query

  • Pushes analytical query processing down to the distributed storage layer
  • Reduces data movement to head node; speeds up large analytical queries
  • Aurora MySQL only

Failover Priority

  • Replicas with lower priority tier number (0 = highest priority) are promoted first
  • Same tier → largest instance type promoted first
  • Failover time: ~30 seconds (vs ~60–120s for RDS Multi-AZ)

Backups & Recovery

  • Automated backups: 1–35 day retention; continuous to S3; zero I/O impact
  • Snapshots: manual; kept until deleted; cross-region copy supported
  • Backtrack (MySQL only): rewind to any point in last 72 hours in-place, without restoring from a snapshot — takes seconds

Aurora vs RDS

StorageAuto-scaling, 6-way replicatedManual EBS provisioning
Read replicasUp to 15, < 10ms replication lagUp to 5, async
Failover~30s (replica promotion)~60–120s (standby promotion)
Standby readableYes (replicas serve reads)No (Multi-AZ standby is hot standby only)
BacktrackYes (MySQL, 72hr window)No
GlobalYes (Global Database)No
Cost~20% more than equivalent RDSLower base cost
  • Aurora replicas serve reads and are automatic failover targets — RDS read replicas are only for reads
  • RDS Multi-AZ standby is not readable — it exists solely for failover
  • Backtrack ≠ restore from snapshot: backtrack is in-place and takes seconds; a snapshot restore creates a new cluster
  • Aurora Serverless v1 → scales to zero; v2 → does not. Know which to recommend for "scale to zero on idle"
