Redis vs Memcached

FeatureRedisMemcached
Data structuresStrings, hashes, lists, sets, sorted sets, bitmaps, HyperLogLog, streamsStrings only
PersistenceRDB snapshots + AOF (append-only file)None — in-memory only
ReplicationYes — primary + up to 5 read replicasNo
Multi-AZ failoverYes — automatic failover to replicaNo
Pub/Sub messagingYesNo
Sorted sets / leaderboardsYesNo
Multi-threadingSingle-threaded (mostly)Multi-threaded — scales horizontally across cores
Cluster modeYes — sharding across nodesYes — but no replication per shard

Redis: Key Capabilities

  • Sub-millisecond latency — ideal for session stores, leaderboards, rate limiters, caching
  • Multi-AZ with automatic failover — primary + replica in separate AZs; failover in ~30–60 seconds; promotes a replica to primary
  • Redis Cluster Mode — data sharded across multiple node groups (shards); each shard has its own primary + replicas; scales both read and write throughput
  • Global Datastore — cross-region active-passive replication; secondary clusters can serve reads; promotes secondary on regional failure
  • Backup & restore — RDB snapshots to S3; restore to a new cluster

Caching Strategies

StrategyHow It WorksTrade-off
Lazy Loading (Cache-aside)App checks cache first; on miss, reads from DB and writes to cacheStale data possible; cold start misses add latency
Write-ThroughApp writes to cache and DB on every writeCache always fresh; extra write latency; cache churn for rarely-read data
Write-Behind (Write-back)Write to cache first; async flush to DBLower write latency; risk of data loss if cache fails before flush
TTL-based expirySet expiry on cached items; auto-evicted after TTLSimple staleness control; tuning TTL is application-specific

Session Store Pattern

  • Store HTTP session data in ElastiCache Redis rather than on the application server
  • Makes application tier stateless — any instance can serve any request
  • Enables clean Auto Scaling and instance replacement without session loss
  • Alternative: DynamoDB for session storage (lower cost, more durable, millisecond vs sub-millisecond)

Common Exam Scenarios

ScenarioAnswer
Sub-millisecond session storage that survives node failureElastiCache Redis with Multi-AZ
Leaderboard / ranked sorted data structureElastiCache Redis sorted sets
Real-time pub/sub messaging between servicesElastiCache Redis pub/sub
Simple cache, multi-threaded, no persistence neededElastiCache Memcached
Offload read traffic from RDSElastiCache Redis (lazy loading or write-through)
Cross-region cache replicationElastiCache Redis Global Datastore
  • Memcached has no replication, no persistence, no failover — a node failure loses all cached data
  • Redis Multi-AZ failover takes ~30–60s — cache-dependent apps must handle cache misses gracefully during this window
  • ElastiCache is deployed in a VPC — not publicly accessible; application and cache must be in the same VPC or connected VPCs
  • Redis Cluster Mode changes the key space distribution — not all Redis commands work across shards (e.g. multi-key operations only work if keys are in the same shard)
