ECS, EKS & Fargate

Container orchestration, launch types, service mesh, auto scaling, App2Container

ECS vs EKS vs Fargate

ECSEKSFargate
OrchestratorAWS-proprietaryKubernetes (upstream)N/A — serverless data plane
Control planeAWS-managedAWS-managedAWS-managed
Data planeEC2 or FargateEC2 or FargateServerless — no nodes to manage
Kubernetes compatibilityNoYes — use existing K8s toolingWorks with both
Best forNew containerised apps on AWS; simpler opsExisting K8s workloads; multi-cloud portabilityEliminating EC2 management overhead

ECS Core Concepts

  • Task Definition: blueprint — container image, CPU/memory, environment variables, volumes, IAM task role
  • Task: running instance of a task definition; ephemeral
  • Service: maintains desired count of tasks; integrates with load balancers; handles replacements on failure
  • Cluster: logical grouping of tasks/services; spans AZs
  • Task Role vs Execution Role: task role = permissions the container code uses (e.g. S3 access); execution role = permissions ECS uses to pull images and write logs

Launch Types

EC2 Launch TypeFargate Launch Type
InfrastructureYou provision and manage EC2 instancesAWS manages the underlying compute
Cost modelPay for EC2 instances (can use Spot for savings)Pay per vCPU + memory per second
ControlFull OS access; customise instance typesNo OS access; fixed isolation per task
Spot supportYes — run tasks on Spot instancesFargate Spot — interrupted with 2-min warning
Best forCost-optimised, GPU workloads, compliance needing dedicated hostsSimplicity, variable workloads, no ops overhead

Auto Scaling

  • Service Auto Scaling: scales ECS/EKS task count based on CPU, memory, ALB request count, or custom CloudWatch metrics
  • Target tracking: simplest — maintain a metric at a target value (e.g. 70% CPU)
  • Step scaling: different responses to different alarm thresholds
  • Cluster Auto Scaling (EC2): ECS Capacity Providers + Auto Scaling Group — scales EC2 nodes when tasks can't be placed

App2Container (A2C)

  • AWS tool that analyses running Java and .NET apps on-premises or EC2
  • Automatically generates Dockerfiles, ECS/EKS task definitions, and CloudFormation/CDK templates
  • Use for containerisation migration strategy (lift-and-shift to containers)

EKS Key Concepts

  • Managed node groups: AWS provisions and manages EC2 workers; supports Spot instances
  • EKS Anywhere: run EKS on-premises on your own hardware
  • EKS Distro: the same Kubernetes distribution AWS uses — run it anywhere
  • AWS Load Balancer Controller: provisions ALB for Ingress, NLB for Service type LoadBalancer
  • Karpenter: open-source node provisioner; more flexible and faster than Cluster Autoscaler

Common Exam Scenarios

ScenarioAnswer
Containerise Java app from on-premises without rewriting itAWS App2Container → ECS or EKS task definition
Run containers without managing serversECS or EKS with Fargate launch type
Migrate existing Kubernetes workload to AWSAmazon EKS (native Kubernetes compatibility)
Reduce container cost for fault-tolerant batch jobsECS with EC2 Spot or Fargate Spot
Run a long-running job that exceeds Lambda 15-min limitECS/Fargate task (no time limit)
Isolate containers at network level with Security GroupsECS awsvpc network mode (one ENI per task)
  • awsvpc mode is required for Fargate — tasks get dedicated ENIs; Security Groups apply per task not per host
  • ECS does not require Kubernetes knowledge — EKS does; choose based on existing tooling and skills
  • Fargate Spot can be interrupted — workloads must handle graceful shutdown on 2-minute warning
  • Task role provides AWS API credentials to the container — do not inject credentials via environment variables; use the task role
