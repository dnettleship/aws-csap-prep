Envelope Encryption

  • KMS keys (Key Encryption Keys / KEK) encrypt data keys, not the data directly
  • Flow: generate a data key (plaintext + encrypted copy) via GenerateDataKey → encrypt data locally with plaintext data key → discard plaintext key → store encrypted data + encrypted data key together
  • To decrypt: call KMS Decrypt with the encrypted data key → KMS returns plaintext data key → decrypt data locally
  • Benefit: data is encrypted locally (fast), KMS only handles small data keys (audit trail, access control)

Key Policy

  • Every KMS key has a resource-based key policy — this is the primary access control mechanism
  • IAM policies alone cannot grant KMS access — key policy must explicitly allow it (or delegate to IAM with "Principal": {"AWS": "arn:aws:iam::ACCOUNT:root"})
  • Grants: temporary, programmatic delegations; used by AWS services on your behalf (e.g. AWS Secrets Manager rotating a secret)
  • Cross-account access: add target account principal to key policy, then target account IAM policy allows the KMS actions

Key Rotation

  • Auto-rotation creates new key material but retains the same Key ID — no re-encryption of existing data needed
  • Old key material is retained to decrypt data encrypted with previous versions
  • Key rotation does not re-encrypt existing ciphertext — you must do that explicitly if required

CloudHSM

  • Dedicated hardware security module (FIPS 140-2 Level 3) in your VPC
  • AWS manages hardware; you manage keys and users — AWS has no access to key material
  • Use when: compliance requires dedicated HSM, or need to export keys, or use custom/non-symmetric algorithms
  • CloudHSM cluster spans AZs for HA; keys synchronised across instances
  • Can be used as a custom key store for KMS — CMKs backed by CloudHSM

AWS Secrets Manager

  • Stores, retrieves, and automatically rotates secrets (DB credentials, API keys, tokens)
  • Automatic rotation uses a Lambda function; built-in support for RDS, Redshift, DocumentDB, Elasticache
  • Secrets encrypted at rest with KMS (AWS managed or CMK)
  • Cross-account access via resource-based policy on the secret
  • Replicate secrets to multiple regions for DR and latency
  • Charges per secret per month + per 10,000 API calls

  • KMS key policy AND IAM policy must both allow — IAM alone is not sufficient
  • KMS encrypt/decrypt API has a 4 KB limit — use envelope encryption for larger data
  • CloudHSM is FIPS 140-2 Level 3; KMS is Level 2 — for compliance requiring Level 3, choose CloudHSM
  • Secrets Manager rotation briefly has two versions of the secret (AWSPENDING, AWSCURRENT) — apps must handle this window
  • Deleting a KMS key has a 7–30 day waiting period — there is no immediate deletion
