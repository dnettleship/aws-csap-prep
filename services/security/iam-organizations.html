IAM, Organizations & SCPs

Policy evaluation logic, SCP patterns with examples, permission boundaries, Identity Center

IAM Policy Evaluation Logic

Evaluated in order — an explicit Deny at any layer wins:

  1. Explicit Deny — in any attached policy; always wins
  2. SCP Allow — must allow if SCPs are applied to the account
  3. Resource-based policy Allow — can grant cross-account access without role assumption
  4. Identity-based policy Allow — must be present
  5. Permission Boundary Allow — must allow if a boundary is set on the principal
  6. Session Policy Allow — must allow if using sts:AssumeRole with an inline session policy

Default is implicit Deny. An action is allowed only if every applicable layer permits it and none deny it.

Cross-Account Resource Access

  • Role assumption — principal calls sts:AssumeRole; receives temporary credentials for target account
  • Resource-based policy — target resource (S3, KMS, SNS, SQS, Lambda) grants access directly to source principal — no role switch needed

Service Control Policies (SCPs)

  • Attached to root, OU, or individual account; children inherit from parents
  • Management account is always exempt — SCPs never apply to it
  • SCPs do not grant permissions; they define the maximum allowed permissions
  • Service-linked roles are not restricted by SCPs

Deny actions outside approved regions

{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "aws:RequestedRegion": ["eu-west-1", "eu-west-2"]
    }
  }
}

Prevent disabling CloudTrail

{
  "Effect": "Deny",
  "Action": ["cloudtrail:StopLogging", "cloudtrail:DeleteTrail"],
  "Resource": "*"
}

Require MFA for IAM and CloudTrail actions

{
  "Effect": "Deny",
  "Action": ["iam:*", "cloudtrail:*"],
  "Resource": "*",
  "Condition": {
    "BoolIfExists": { "aws:MultiFactorAuthPresent": "false" }
  }
}

IAM Identity Center (SSO)

  • Centralised access management across all accounts in the organisation
  • Permission Sets are deployed as IAM roles in each member account
  • Integrates with Managed AD, AD Connector, or external SAML 2.0 IdP
  • SCIM protocol for automated user provisioning from IdP

Permission Boundaries

  • An IAM managed policy attached to a user or role that caps their maximum permissions
  • Use case: allow developers to create IAM roles without enabling privilege escalation — the boundary ensures new roles cannot exceed the developer's own permissions
  • Boundary + identity policy: effective permissions = intersection of both
  • SCP Allow at root + SCP Deny at OU = Deny wins for all accounts in that OU
  • If no SCP is attached to an account, the default is FullAWSAccess (implicit allow all at the SCP layer)
  • aws:PrincipalOrgID condition key — restrict resource access to principals from your org only
  • Role chaining: max session duration is 1 hour regardless of individual role settings
