Monitoring

CloudWatch & X-Ray

Metrics, alarms, logs, dashboards, Container Insights, distributed tracing

CloudWatch Metrics

  • Standard metrics: AWS service metrics published automatically (EC2 CPU, RDS FreeStorageSpace, etc.)
  • Detailed monitoring: EC2 metrics every 1 minute (vs 5-min default); extra cost
  • Custom metrics: publish application metrics via PutMetricData API; standard (1-min) or high-resolution (1-second)
  • Metric Math: combine multiple metrics into expressions (e.g. error rate = errors / requests)
  • Anomaly Detection: ML model learns metric baseline; alarm when metric is outside expected band

CloudWatch Alarms

  • States: OK, ALARM, INSUFFICIENT_DATA
  • Actions: SNS notification, Auto Scaling policy, EC2 action (stop/terminate/reboot/recover), Systems Manager OpsItem
  • Composite alarms: combine multiple alarms with AND/OR logic; reduces alarm noise
  • Math alarms: alarm on a metric expression, not just a raw metric
  • Evaluation period + datapoints to alarm: e.g. "3 out of 5 datapoints above threshold" — reduces false positives from spikes

CloudWatch Logs

  • Log groups: container for log streams; set retention (1 day – 10 years or never expire)
  • Metric filters: extract metric values from log content (e.g. count of ERROR occurrences)
  • CloudWatch Logs Insights: interactive SQL-like query language over log data; query multiple log groups; supports JSON log parsing
  • Subscription filters: stream log data in near-real-time to Kinesis, Lambda, or OpenSearch
  • Cross-account log sharing: share logs across accounts using Logs resource policies
  • Log encryption: optionally encrypt log groups with KMS CMK

CloudWatch Container Insights

  • Collects metrics and logs from ECS, EKS, and Kubernetes clusters
  • Provides pre-built dashboards: cluster, service, task/pod CPU, memory, network
  • For EKS: requires the CloudWatch agent deployed as a DaemonSet
  • For ECS (EC2): CloudWatch agent sidecar; Fargate: built-in support via task metadata

CloudWatch Dashboards

  • Cross-account, cross-region dashboards — single pane of glass for multi-account environments
  • Shareable via link (public or requiring AWS sign-in); embed in internal portals

AWS X-Ray (Distributed Tracing)

  • Traces requests end-to-end across microservices; identifies latency bottlenecks and errors
  • Segments: data for one service; subsegments: breakdown within a segment (e.g. DynamoDB call duration)
  • Service Map: visual graph of service dependencies and health; shows latency and error rates per edge
  • Sampling: records a configurable fraction of requests (default 5%/second + 5% of remainder) — reduces cost; configurable rules
  • Integrations: Lambda (built-in active tracing toggle), API Gateway, ECS, EC2 (X-Ray daemon), Elastic Beanstalk

VPC Flow Logs

  • Captures metadata about network traffic to/from ENIs in a VPC; not packet captures
  • Publish to CloudWatch Logs or S3
  • Use for: security analysis, identifying unexpected traffic, troubleshooting connectivity
  • Captured fields: srcaddr, dstaddr, srcport, dstport, protocol, packets, bytes, action (ACCEPT/REJECT)

CloudTrail

  • Records all AWS API calls (who did what, when, from where)
  • Management events: control-plane operations (create/delete/modify resources) — on by default in all regions
  • Data events: S3 object-level operations, Lambda invocations — not enabled by default; extra cost
  • Deliver to S3 and/or CloudWatch Logs; cross-account via organization trail
  • Integrity validation: hash digest files detect tampering

Common Exam Scenarios

ScenarioAnswer
Alert when error count in application logs exceeds thresholdCloudWatch Logs metric filter → CloudWatch Alarm → SNS
Identify which Lambda invocation caused high latencyAWS X-Ray with active tracing enabled on Lambda
Monitor CPU/memory for ECS containersCloudWatch Container Insights
Query logs across 10 services for a specific request IDCloudWatch Logs Insights (multi-log-group query)
Detect unusual metric patterns without a fixed thresholdCloudWatch Anomaly Detection
Who deleted the S3 bucket? Audit API callsAWS CloudTrail — management events
Stream security logs in real-time to SIEMCloudWatch Logs subscription filter → Kinesis Data Firehose → SIEM
  • CloudWatch metrics default retention varies by period — 1-second metrics kept 3 hours; 1-minute kept 15 days; 5-minute kept 63 days; 1-hour kept 15 months
  • CloudTrail records API calls, not OS-level activity — for OS/application logging, use CloudWatch Logs agent or SSM
  • VPC Flow Logs capture metadata only — packet payloads are not captured
  • X-Ray sampling means not every request is traced by default — for 100% sampling, configure a custom sampling rule
  • CloudWatch Alarms have a minimum 10-second resolution with high-resolution custom metrics
