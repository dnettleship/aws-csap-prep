SQS, SNS & EventBridge

Decoupling patterns, fan-out, event routing, FIFO ordering, DLQs

SQS — Simple Queue Service

Standard vs FIFO

Standard QueueFIFO Queue
OrderingBest-effort (not guaranteed)Strict first-in-first-out per message group
DeliveryAt-least-once (duplicates possible)Exactly-once processing (deduplication)
ThroughputUnlimited3,000 msg/s with batching; 300 msg/s without
DeduplicationConsumer must handle duplicatesDeduplication ID; 5-min dedup window
Message groupsNoMessageGroupId for parallelism within FIFO

Key SQS Concepts

  • Visibility timeout — when a consumer receives a message, it becomes invisible for this duration (default 30s); if not deleted, it reappears for another consumer; set to > max processing time
  • Dead-Letter Queue (DLQ) — messages that fail processing N times (maxReceiveCount) are moved here; use for debugging and isolating poison pill messages
  • Long polling — consumer waits up to 20s for messages; reduces empty responses and cost vs short polling
  • Message retention — 1 minute to 14 days (default 4 days)
  • Max message size — 256 KB; use the Extended Client Library for larger payloads (stores in S3)
  • Delay queues — delay message delivery by up to 15 minutes at queue or per-message level

SNS — Simple Notification Service

  • Pub/sub: one publisher, many subscribers; push-based (no polling)
  • Subscribers: SQS, Lambda, HTTP/HTTPS, email, SMS, mobile push, Kinesis Data Firehose
  • Fan-out pattern: SNS topic → multiple SQS queues; decouple publisher from multiple consumers
  • Filter policies: subscriber-level JSON filters; each subscriber only receives matching messages (reduces processing cost)
  • SNS FIFO: ordered delivery + deduplication; only SQS FIFO queues as subscribers
  • Message archiving & replay: store messages in S3 for compliance and reprocessing
  • DLQ support: per-subscription DLQ for failed deliveries

EventBridge

  • Serverless event bus; routes events from AWS services, SaaS providers, and custom applications
  • Event buses: default (AWS service events), custom (application events), partner (SaaS events)
  • Rules: match events by pattern (JSON path conditions); send to one or more targets; up to 5 targets per rule
  • Targets: Lambda, SQS, SNS, Step Functions, Kinesis, API GW, ECS tasks, EventBridge bus in another account
  • Schema Registry: auto-discovers event schemas; generates code bindings for type-safe handling
  • Pipes: point-to-point connections with optional filtering and enrichment (source → filter → enrich → target)
  • Scheduling: EventBridge Scheduler replaces CloudWatch Events rules for cron/rate-based scheduling

Choosing the Right Service

RequirementUse
Decouple producer from consumer; consumer pulls at own paceSQS
Guaranteed ordering + exactly-once per financial transactionSQS FIFO
One event → multiple different consumers in parallelSNS fan-out → multiple SQS queues
Route events from SaaS tools (Datadog, Zendesk) to LambdaEventBridge partner event bus
Complex event routing rules that change frequentlyEventBridge rules
Buffer writes to protect a downstream databaseSQS → Lambda → RDS
Trigger Lambda on S3 object creation (event-driven)S3 Event → SNS or EventBridge or direct Lambda trigger

Common Patterns

  • Fan-out: S3 → SNS → [SQS A, SQS B, Lambda C] — one upload triggers parallel processing pipelines
  • Worker pool: SQS → Auto Scaling Group (scale on queue depth) — handles bursty jobs
  • Cross-account events: EventBridge resource policy allows another account to put events on a bus
  • Saga / orchestration: Step Functions for multi-step workflows with SQS/SNS between steps for loose coupling
  • SQS Standard can deliver messages out of order and more than once — consumers must be idempotent
  • FIFO queues have a throughput limit — Standard queues if you need unlimited throughput; implement ordering in the consumer
  • SNS cannot deliver to SQS in a different account without an SQS resource policy granting SNS permission
  • EventBridge event buses do not retain messages — if no rule matches, the event is dropped
  • Visibility timeout must be longer than consumer processing time — otherwise the same message is processed twice
