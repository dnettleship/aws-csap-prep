Routing Policies

PolicyBehaviourUse Case

WeightedTraffic split by configured weight (0–255); weight 0 = no trafficCanary deploys, A/B testing, gradual migration
LatencyRoutes to region with lowest measured latency for the userActive-active multi-region; global performance
FailoverActive-passive; health check monitors primary; automatic cutover to secondary on failureDR primary/secondary setup

GeoproximityRoutes by geographic distance; bias value shifts the boundaryFine-grained regional traffic shifting (requires Traffic Flow)

Health Checks

  • Endpoint health checks — poll an HTTP/HTTPS/TCP endpoint every 10 or 30 seconds; threshold of consecutive failures triggers unhealthy state
  • Calculated health checks — combine multiple child health checks with AND/OR logic; useful for composite service health
  • CloudWatch alarm health checks — mark unhealthy when a CloudWatch alarm enters ALARM state; use for private resources that can't be polled directly
  • Health checks used with Failover routing determine when to cut over to the secondary
  • Health checks used with Weighted / Latency / Multivalue remove unhealthy records from responses
  • Propagation time: ~30–60 seconds from unhealthy detection to DNS propagation — factor into RTO

Private Hosted Zones

  • DNS resolution within a VPC — not visible over the internet
  • Must enable DNS Resolution and DNS Hostnames on the VPC
  • Associate with multiple VPCs (including cross-account VPCs)
  • Split-horizon DNS: same domain name resolves differently inside vs outside a VPC

DNSSEC

  • Route 53 supports DNSSEC signing for public hosted zones
  • Protects against DNS spoofing / cache poisoning
  • Uses KMS asymmetric keys for signing

Automatic DNS failover to DR regionFailover routing + health check on primary
Send 10% of traffic to new app versionWeighted routing (10 / 90 weights)
Restrict EU users to EU-region endpoint (data sovereignty)Geolocation routing by continent
DNS record at zone apex pointing to ALBAlias record (CNAME not allowed at apex)
Health check for resource in private subnet (no public endpoint)CloudWatch alarm health check
Gradual traffic shift between regions with geographic boundary controlGeoproximity routing with bias
  • Route 53 is a global service — hosted zones are not region-specific
  • Failover routing requires health checks — without them it always points to primary
  • Geolocation routing requires a default record — traffic with no matching geolocation returns NXDOMAIN otherwise
  • Latency routing is based on AWS-measured latency, not geographic distance — geographically close regions can have different latency rankings
  • Simple routing with multiple values returns all IPs — client picks one; no health check support
