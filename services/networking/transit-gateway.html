AWS Transit Gateway

Regional virtual router — hub-and-spoke topology replacing VPC peering meshes

What It Is

Regional virtual router that connects VPCs and on-premises networks in a hub-and-spoke topology. Each new spoke only needs one attachment — no O(n²) peering connections.

Key Concepts

  • Attachments — VPC, VPN, Direct Connect Gateway, TGW Connect (SD-WAN/GRE), inter-region TGW peering
  • Route Tables — each attachment is associated with exactly one route table; multiple route tables enable traffic segmentation
  • Propagation — attachments can automatically propagate their CIDR routes into a route table
  • Associations — each attachment has exactly one association (which route table controls its outbound routing)

Connectivity Patterns

Hub-and-Spoke (default)

All VPCs attach to TGW → single shared route table → full mesh connectivity between all spokes.

Segmented / Isolated Routing

  • Create separate route tables per environment (prod, dev, shared-services)
  • Prod VPCs: associated with prod-rt; only shared-services propagates into prod-rt → prod can reach shared services but not dev
  • Use case: compliance, blast radius reduction

Inter-Region

  • TGW peering attachments connect TGWs across regions
  • Static routes required across peering (no dynamic propagation)
  • Traffic stays entirely on the AWS global backbone

Limits & Bandwidth

  • Max bandwidth per VPC attachment: 50 Gbps burst
  • Max attachments per TGW: 5,000
  • ECMP across VPN attachments: add multiple tunnels to scale VPN throughput beyond 1.25 Gbps

TGW vs VPC Peering

Cross-accountYes (share via RAM)Yes
  • VPC peering is non-transitive — A↔B and B↔C does not enable A↔C. TGW solves this.
  • RAM is required to share a TGW across accounts in an organisation
  • TGW Connect enables BGP over GRE for SD-WAN appliances — higher throughput than Site-to-Site VPN
  • Appliance Mode: ensures symmetric routing for stateful NVAs (firewalls/IDS) in a shared-services VPC
